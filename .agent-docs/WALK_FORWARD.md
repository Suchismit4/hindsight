# Walk-Forward Analysis

The walk-forward system splits datasets into temporal segments (train/infer pairs), fits processors per training window, transforms inference data using learned states, and optionally trains models per segment. It enforces temporal isolation to prevent lookahead bias.

---

## SegmentConfig Fields

`SegmentConfig` (in `src/pipeline/walk_forward/segments.py`) defines parameters for generating a rolling walk-forward schedule:

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `start` | `np.datetime64` | Yes | First timestamp to consider for scheduling |
| `end` | `np.datetime64` | Yes | Last timestamp to consider for scheduling |
| `train_span` | `np.timedelta64` | Yes | Length of each training window |
| `infer_span` | `np.timedelta64` | Yes | Length of each inference window |
| `step` | `np.timedelta64` | Yes | Shift between consecutive segments (typically = `infer_span`) |
| `gap` | `np.timedelta64` | No (default `0s`) | Gap between `train_end` and `infer_start` to avoid leakage |
| `clip_to_data` | `bool` | No (default `True`) | Clip segments to actual data availability |

**Validation rules:**

- `start < end`
- `step > 0`, `train_span > 0`, `infer_span > 0`
- `gap >= 0` (though the system doesn't enforce non-negative explicitly, negative gap would overlap train and infer)

**In YAML specs** (`model.walk_forward`), these are specified in hours:

```yaml
walk_forward:
  train_span_hours: 720     # 30 days
  infer_span_hours: 24      # 1 day
  step_hours: 24             # 1 day step
  gap_hours: 0               # no gap
  start: "2023-01-01T00:00:00"   # optional
  end: "2023-12-31T23:00:00"     # optional
```

## Segment and SegmentPlan

A `Segment` defines one train/infer pair:

```python
@dataclass(frozen=True)
class Segment:
    train_start: np.datetime64   # inclusive
    train_end: np.datetime64     # inclusive
    infer_start: np.datetime64   # inclusive
    infer_end: np.datetime64     # exclusive
```

**Boundary convention:**

- Training: `[train_start, train_end]` — inclusive both ends
- Inference: `[infer_start, infer_end)` — exclusive at end to prevent overlap with next segment

A `SegmentPlan` is an ordered collection of `Segment` objects with validation utilities:

```python
class SegmentPlan:
    segments: List[Segment]
    def validate(self, allow_overlaps=False) -> List[str]
    def total_inference_period -> (start, end)
    def total_training_period -> (start, end)
```

Plans are generated by `make_plan(config: SegmentConfig)` in `src/pipeline/walk_forward/planning.py`.

## WalkForwardRunner vs ModelRunner

Both runners follow the same gather→scatter pattern but serve different purposes:

| Aspect | WalkForwardRunner | ModelRunner |
|--------|-------------------|-------------|
| **Location** | `src/pipeline/walk_forward/execution.py` | `src/pipeline/model/runner.py` |
| **Purpose** | Data processing only — no models | Data processing + model training/prediction |
| **Output** | `WalkForwardResult` (processed xr.Dataset) | `ModelRunnerResult` (predictions xr.Dataset) |
| **Model integration** | None | Takes `model_factory` → fits/predicts per segment |
| **State** | Returns per-segment processor states | Returns per-segment model + processor states |
| **Progress bar** | `"Walk-forward analysis"` | `"Walk-forward segments"` |

**Shared execution strategy (both runners):**

1. Compute shared view once (`_ensure_shared()`)
2. Pre-stack shared dataset for fast integer slicing
3. Precompute segment bounds via `_compute_bounds()` using `np.searchsorted`
4. For each segment:
   - Slice train/infer from stacked shared, unstack
   - Fit learn processors on train → get states
   - Apply learned states to infer (transform-only)
   - Apply infer processors (transform-only)
   - *(ModelRunner only)* Fit model on processed train, predict on processed infer
5. Scatter results into global buffer using `_masked_scatter_rows_inplace()`
6. Unstack final buffer back to `(year, month, day, hour, asset)`

**Scatter policies:**

| Policy | Behavior |
|--------|----------|
| `"last"` | Overwrite buffer where new segment has non-NaN values |
| `"first"` | Write only into currently-NaN cells where new segment has non-NaN values |

## Leakage Contract

The walk-forward system enforces strict temporal isolation:

### What the learn stage CAN see

- **Training data only**: the dataset slice from `train_start` to `train_end`
- **Shared-processed features**: the output of shared processors applied to the training slice
- **Its own learned states**: each learn processor fits on the train slice and produces state

### What the learn stage CANNOT see

- **Inference data**: the infer slice is never passed to `fit()`
- **Future segments**: each segment is processed independently
- **Global statistics**: learn processors see only the current training window, not the full dataset
- **Other segments' learned states**: each segment produces fresh states

### How isolation is enforced

1. `_compute_bounds()` uses `np.searchsorted` on the stacked time vector to compute exact integer slices for each segment
2. Train and infer data are sliced from the pre-stacked shared view using `isel(time_index=slice)` — no overlap
3. `handler._apply_pipeline(train_ds, pipeline=config.learn, fit=True)` fits processors only on `train_ds`
4. Learned states are applied to `infer_ds` via `proc.transform(infer_ds, state)` — transform-only, no fitting
5. The `gap` parameter in `SegmentConfig` adds explicit separation between train end and infer start

### Common leakage pitfalls to avoid

| Pitfall | Prevention |
|---------|------------|
| Using `DataHandler.build()` directly | Use `WalkForwardRunner` or `ModelRunner` instead — they handle per-segment fitting |
| Fitting on the full dataset before walk-forward | Shared processors are transform-only; learn processors are fit per segment |
| Normalizing with global mean/std | `CSZScore` in learn stage fits per segment; global stats would leak future information |
| Forward-filling across segment boundaries | `PerAssetFFill` operates within the slice provided; the runner handles slicing |

## ModelAdapter Interface for Custom Models

```python
# src/pipeline/model/adapter.py
class ModelAdapter:
    def fit(self, ds, features, label=None, sample_weight=None) -> "ModelAdapter":
        """Train the model. Returns self for chaining."""
        raise NotImplementedError

    def partial_fit(self, ds, features, label=None, sample_weight=None) -> "ModelAdapter":
        """Optional incremental training. Default falls back to fit."""
        return self.fit(ds, features, label, sample_weight)

    def predict(self, ds, features) -> xr.DataArray:
        """Generate predictions. Returns DataArray named by output_var."""
        raise NotImplementedError

    def get_state(self) -> Optional[Dict[str, Any]]:
        """Optional: return JSON-serializable model state for persistence."""
        return None

    def load_state(self, state: Dict[str, Any]) -> None:
        """Optional: restore state from get_state()."""
        return None
```

### SklearnAdapter (built-in)

```python
@dataclass
class SklearnAdapter(ModelAdapter):
    model: Any                  # sklearn estimator with .fit(X, y) and .predict(X)
    handler: DataHandler        # Used for xarray → numpy stacking
    output_var: str = "score"   # Name of prediction variable
    use_proba: bool = False     # Use predict_proba instead of predict
    proba_index: int = 1        # Column of predict_proba to use
    drop_nan_rows: bool = True  # Drop rows with non-finite features
```

**Data conversion flow:**

1. `handler.to_arrays_for_model(ds, features, labels)` → `X: (T, N, J)`, `y: (T, N, K)`
2. Flatten to 2D: `X2: (T*N, J)`, `y2: (T*N,)`
3. Drop rows where any feature or label is non-finite
4. `model.fit(X2_clean, y2_clean)` (with optional `sample_weight`)
5. `model.predict(X2)` → reshape back to `(T, N)` → `xr.DataArray`

### How to implement a custom adapter

```python
class MyModelAdapter(ModelAdapter):
    def __init__(self, model, handler, output_var="pred"):
        self.model = model
        self.handler = handler
        self.output_var = output_var

    def fit(self, ds, features, label=None, sample_weight=None):
        X, y = self.handler.to_arrays_for_model(ds, features, [label])
        # Custom training logic
```

The adapter is passed to `ModelRunner` via `model_factory`:

```python
runner = ModelRunner(
    handler=handler,
    plan=plan,
    model_factory=lambda: MyModelAdapter(MyModel(), handler, "pred"),
    feature_cols=["feat1", "feat2"],

```

A fresh adapter is instantiated per segment to prevent state leakage between training windows.
